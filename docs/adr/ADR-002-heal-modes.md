# ADR-002: Auto-Heal 三级执行模式

## 状态
已采纳

## 背景
Auto-heal 功能能够自动诊断和修复生产故障，但自动修改代码和部署存在高风险：
- 错误的自动修复可能引入新 bug
- 未经审核的代码变更违反合规要求
- 生产环境的自动部署可能导致服务中断

## 决策
设计三级执行模式，逐步放开自动化程度：

### propose-only（默认）
- 仅诊断和建议，不修改任何文件
- 输出标记 `dry_run: true`
- 适用于所有环境，零风险

### semi-auto
- 在 propose-only 基础上，自动创建 Git 分支并提交修复代码
- 不自动合并，需人工审批
- 适用于有 CI/CD 流程的团队

### full-auto
- 自动应用修复、运行测试、部署
- 测试失败自动回滚
- 硬性限制：`OPSZEN_ENV=production` 时拒绝执行
- 仅适用于开发/测试环境

## 安全护栏
- 频率限制（默认 3 次/小时）防止修复风暴
- 审计日志记录完整操作链路
- 备份机制支持任意时间点回滚
- 过期备份自动清理

## 考虑的替代方案

1. **仅 propose-only**：最安全但价值有限，无法体现自动化优势。
2. **仅 full-auto**：风险过高，不适合生产环境。
3. **基于审批的工作流**：需要外部审批系统集成，增加架构复杂度。

## 结论
三级模式让用户根据环境和风险偏好自主选择自动化程度。默认最保守模式确保安全，同时保留逐步升级的路径。
